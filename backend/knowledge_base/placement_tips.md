## 1. Introduction

## 2. Resume Writing (ATS-Friendly)

### 2.1 What is ATS?

An **Applicant Tracking System (ATS)** is software used by recruiters to filter resumes before a human reads them. Studies show that **75% of resumes are rejected by ATS** before reaching a recruiter. Your resume must be machine-readable first, human-impressive second.

---

### 2.2 ATS Optimization Tips

| ATS Rule | Do This | Avoid This |
|---|---|---|
| Keywords | Match job description terms exactly | Using synonyms that don't match JD |
| File Format | Submit as `.pdf` or `.docx` | `.jpg`, `.png`, `.pages` formats |
| Fonts | Use standard fonts (Arial, Calibri, Times New Roman) | Decorative or custom fonts |
| Tables & Graphics | Use plain text and bullet points | Tables, charts, icons, images |
| Headers | Use standard section headings | Creative names like "My Story" or "Where I've Been" |
| Columns | Use single-column layout | Multi-column layouts that confuse parsers |
| Special Characters | Use hyphens and bullets | Unicode arrows, emojis, or decorative symbols |
| File Name | `FirstName_LastName_Resume.pdf` | `Resume_Final_v2_ACTUAL.pdf` |

#### Keyword Strategy

- Read the **job description carefully** and identify technical skills, tools, and soft skills mentioned.
- Incorporate these keywords **naturally** in your Experience, Skills, and Summary sections.
- Use both acronyms and full forms: `ML (Machine Learning)` in your skills section.
- For software roles, include: programming languages, frameworks, databases, tools, platforms.

### 2.4 The 1-Page Rule & Exceptions

**Rule:** For freshers and students with less than 2 years of experience, **your resume must be exactly 1 page.**

**Why?** Recruiters spend an average of **6–7 seconds** on first glance. A concise, well-structured 1-pager shows clarity of thought.

| Scenario | Recommended Length |
|---|---|
| Fresher / Student (0–1 year) | 1 Page — Non-negotiable |
| 1–3 Years Experience | 1 Page (max 1.5 if needed) |
| 3–7 Years Experience | 2 Pages |
| 7+ Years / Research Profile | 2–3 Pages |
| Academic CV (PhD, Research) | As long as needed |

**Tips to fit in 1 page:**
- Use 10–11pt font for body, 13–14pt for name
- Set margins to 0.5 inches on all sides
- Remove personal pronoun ("I", "My") from all bullet points
- Merge similar experiences if needed
- Remove irrelevant sections (e.g., hobbies unless directly relevant)

### 2.5 Required Resume Sections

A strong fresher resume must contain these sections **in this order:**

#### 1. Contact Information
```
Full Name | City, State | Phone | Email | LinkedIn URL | GitHub URL | Portfolio (optional)
```
- Use a **professional email**: `firstname.lastname@gmail.com`
- LinkedIn URL should be **customized**: `linkedin.com/in/firstname-lastname`
- GitHub should showcase active repositories with README files

#### 2. Professional Summary (3–4 lines)
- Who you are (major, college, graduation year)
- What you're skilled in (key technologies)
- What you're seeking (role type)

**Example:**
> Final-year B.Tech CSE student at LPU with hands-on experience in full-stack web development using React.js and Node.js. Built 3 end-to-end projects involving REST APIs, MongoDB, and AWS deployment. Seeking a Software Development Engineer role to contribute to scalable backend systems.

#### 3. Education
- Degree, University, CGPA, Graduation Year
- Include relevant coursework if CGPA is below 7.5
- List 12th and 10th marks if above 80%

#### 4. Technical Skills
Organize by category:
```
Languages:     Python, Java, C++, JavaScript
Frameworks:    React.js, Node.js, Spring Boot, Django
Databases:     MySQL, MongoDB, Redis
Tools & Platforms: Git, Docker, AWS, Linux, Postman
Concepts:      Data Structures, Algorithms, OOP, REST APIs
```

#### 5. Experience / Internships
- List in **reverse chronological order**
- For each: Company | Role | Duration | Location (Remote/On-site)
- 3–5 bullet points per experience using the STAR format
- Quantify every achievement

#### 6. Projects (2–4 projects)
- Project Name | Tech Stack | Duration | GitHub Link
- Describe: What it does, how you built it, key impact/metrics
- Prioritize projects with real users, open-source contributions, or unique tech

#### 7. Achievements & Certifications
- Competitive programming ratings (Codeforces, LeetCode, CodeChef)
- Hackathon wins or notable rankings
- Google, AWS, Microsoft certifications
- Academic awards or scholarships

### 2.6 How to Quantify Achievements (STAR in Resume)

Quantifying achievements transforms weak bullets into powerful statements.

**Formula:** `Action Verb + Task + Result (with number)`

| Weak Statement | Strong (Quantified) Statement |
|---|---|
| "Built a web app" | "Built a full-stack e-commerce app serving 500+ daily active users with 99.9% uptime" |
| "Improved database queries" | "Optimized SQL queries reducing average response time by 65% (from 800ms to 280ms)" |
| "Worked on ML model" | "Developed a sentiment analysis model achieving 91% accuracy on 50,000+ reviews" |
| "Led a team" | "Led a 5-member team to deliver a hackathon project in 24 hours, winning 1st place among 120 teams" |
| "Fixed bugs" | "Resolved 30+ production bugs reducing customer complaints by 40% over one quarter" |

**Numbers to use:**
- Percentage improvement (speed, accuracy, reduction)
- Scale (users, records, requests per second)
- Time saved (hours/week, days to completion)
- Team size, event size, competition rank
- Lines of code, test coverage percentage

### 2.7 Common Resume Mistakes Table

| Mistake | Why It's Bad | Fix |
|---|---|---|
| Typos and grammatical errors | Signals lack of attention to detail | Use Grammarly + have 3 people proofread |
| Using "Responsible for..." | Passive, weak, tells nothing about impact | Replace with action verb + result |
| Including a photo | Can introduce bias; not standard in India's top companies | Remove photo from resume |
| Listing every skill you've "heard of" | Interviewer may ask deep questions on anything listed | Only list skills you can defend in an interview |
| Generic objective statement | "To work in a dynamic environment..." says nothing | Write a targeted, specific professional summary |
| Not including GitHub/Portfolio | Misses chance to showcase real work | Always include active GitHub with good READMEs |
| Using same resume for all jobs | ATS misses keyword matches | Tailor resume per JD (at least the summary and skills) |
| Irrelevant personal information | Wastes space and is unprofessional | Remove DOB, religion, marital status, father's name |
| Poor formatting / inconsistent fonts | Hard to read, looks unprofessional | Use a consistent template (Jake's Resume, Novoresume) |
| Listing duties instead of achievements | Shows what you did, not what you accomplished | Convert every bullet to achievement with metric |

## 3. LeetCode Strategy

### 3.1 The 5-Step Problem-Solving Framework

Use this structured approach every time you face a new problem:

**Step 1 — Understand the Problem (5 min)**
- Read the problem statement 2–3 times carefully
- Identify: Input type, Output type, Constraints
- Ask: Edge cases? Empty input? Negative numbers? Large N?
- Paraphrase the problem in your own words

**Step 2 — Think of Examples (3 min)**
- Work through the provided examples manually
- Create 2–3 of your own test cases including edge cases
- Draw diagrams, trees, or graphs if the problem involves them

**Step 3 — Identify the Pattern (5 min)**
- Map the problem to a known pattern (see Section 3.3)
- Think: Brute force first → Optimize
- Consider time/space complexity tradeoffs

**Step 4 — Code the Solution (15–20 min)**
- Write clean, readable code with meaningful variable names
- Start with the brute force; optimize after correctness
- Add comments for complex logic blocks

**Step 5 — Test & Optimize (5–7 min)**
- Run through test cases mentally before submitting
- Check for off-by-one errors, null checks, integer overflow
- Analyze time complexity: Can it pass within constraints?

---

### 3.2 When to Look at Hints

| Time Spent | What to Do |
|---|---|
| 0–10 min | Think independently, don't look at hints |
| 10–20 min | Revisit Step 1 and Step 2; re-read carefully |
| 20–30 min | Look at the hint section or discussion title (not full solution) |
| 30–40 min | Look at a high-level approach description only |
| 40+ min | Read the full solution, understand it deeply, then implement from scratch |

> **Key Rule:** Never copy-paste a solution. Always re-implement it yourself from memory. If you can't, read it again and try once more.

### 3.4 Mock Interview Practice

Mock interviews are **non-negotiable**. Solving problems silently at home is very different from explaining your approach under pressure.

**How to do mock interviews:**

1. **Pramp / Interviewing.io** — Free peer-to-peer mock interviews. Schedule 2–3 per week in the final month.
2. **LeetCode Contest** — Participate in weekly contests every Sunday. Aim for solving at least 2/4 problems.
3. **Record yourself** — Use your phone to record a mock interview. Watch it back to catch filler words ("umm", "like") and poor explanation patterns.
4. **Partner with a friend** — Alternate roles (interviewer/interviewee). Interviewer asks follow-up questions.

**Mock interview checklist:**
- [ ] Think aloud throughout
- [ ] State assumptions before coding
- [ ] Discuss time/space complexity before and after coding
- [ ] Test with edge cases verbally
- [ ] Ask clarifying questions at the start

### 4.2 Core Subject Revision Strategy

#### Operating Systems (OS)
**Key Topics:**
- Process vs Thread, Multithreading, Context Switching
- CPU Scheduling Algorithms: FCFS, SJF, Round Robin, Priority
- Memory Management: Paging, Segmentation, Virtual Memory
- Deadlock: Conditions, Prevention, Avoidance (Banker's Algorithm)
- Synchronization: Mutex, Semaphore, Race Condition, Critical Section
- File Systems: FAT, NTFS, Inodes

**Revision Tip:** Draw diagrams for scheduling algorithms. Practice numerical problems for Gantt charts and page replacement (FIFO, LRU, Optimal).

#### Database Management Systems (DBMS)
**Key Topics:**
- ER Diagrams, Normalization (1NF, 2NF, 3NF, BCNF)
- SQL: Joins (INNER, LEFT, RIGHT, FULL), Aggregations, Subqueries, Window Functions
- ACID Properties, Transactions, Isolation Levels
- Indexing: B-Tree, Clustered vs Non-Clustered
- NoSQL vs SQL: When to use which?
- Stored Procedures, Triggers, Views

**Revision Tip:** Practice 50+ SQL queries on LeetCode SQL problems. Understanding JOIN behavior is critical.

#### Computer Networks (CN)
**Key Topics:**
- OSI Model (7 layers) and TCP/IP Model (4 layers)
- TCP vs UDP: differences, use cases
- HTTP vs HTTPS, REST APIs, WebSockets
- DNS, DHCP, ARP, NAT, MAC vs IP Address
- Subnetting, CIDR Notation
- Routing Protocols: OSPF, BGP, RIP

**Revision Tip:** Use the OSI layer mnemonic: "Please Do Not Throw Sausage Pizza Away" (Physical, Data Link, Network, Transport, Session, Presentation, Application).

#### Object-Oriented Programming (OOP)
**Key Topics:**
- Four Pillars: Encapsulation, Abstraction, Inheritance, Polymorphism
- Method Overloading vs Overriding
- Abstract Classes vs Interfaces
- Design Patterns: Singleton, Factory, Observer, Strategy, Decorator
- SOLID Principles
- Static vs Dynamic Binding

**Revision Tip:** Prepare a real-world example for each OOP concept. Interviewers love hearing: "In my project, I used Inheritance when..."

## 5. System Design Basics for Freshers

> System design interviews are primarily for experienced candidates (2+ years), but **freshers at top product companies** are often asked basic HLD questions. LPU students targeting FAANG/MAANG should be comfortable with these concepts.

### 5.1 High-Level Design (HLD) Components

| Component | Role | Examples |
|---|---|---|
| **Load Balancer** | Distribute traffic across multiple servers | Nginx, AWS ELB, HAProxy |
| **Web Server** | Handle HTTP requests | Apache, Nginx |
| **Application Server** | Business logic processing | Node.js, Django, Spring Boot |
| **Database** | Persistent data storage | MySQL (SQL), MongoDB (NoSQL) |
| **Cache** | Fast temporary data storage | Redis, Memcached |
| **Message Queue** | Async communication between services | Kafka, RabbitMQ, SQS |
| **CDN** | Serve static assets globally | Cloudflare, AWS CloudFront |
| **API Gateway** | Single entry point for APIs | AWS API Gateway, Kong |
| **Search Engine** | Full-text search | Elasticsearch, Solr |
| **Object Storage** | Store large files (images, videos) | AWS S3, Google Cloud Storage |

---

### 5.2 Common Systems to Know

#### 🔗 URL Shortener (like bit.ly)
**Requirements:** Take long URL → generate short code → redirect on access
**Key Components:**
- Base62 encoding for short code generation
- Database table: `short_code → original_url`
- Cache (Redis) for frequently accessed URLs
- Handle 301 vs 302 redirects (caching behavior)
- Rate limiting to prevent abuse

#### 💬 Chat Application (like WhatsApp)
**Requirements:** Real-time messaging, online/offline status, media sharing
**Key Components:**
- WebSocket connections for real-time messaging
- Message queue (Kafka) for delivery guarantees
- Database: One for users, one for messages (with TTL for old messages)
- Push notifications for offline users (FCM/APNs)
- End-to-end encryption concepts

#### 🅿️ Parking Lot System
**Requirements:** Multiple levels, different vehicle types, payment system
**Key Components (OOP Design):**
- Classes: ParkingLot, Level, ParkingSpot, Vehicle, Ticket, Payment
- Spot types: Compact, Large, Motorcycle, Handicapped
- Occupancy tracking with real-time updates
- Fare calculation based on entry/exit time

### 5.3 Key Concepts to Know

| Concept | What It Solves | Quick Definition |
|---|---|---|
| **Horizontal Scaling** | Handles increased traffic | Add more servers (scale out) |
| **Vertical Scaling** | Quick performance boost | Add more CPU/RAM to one server (scale up) |
| **Load Balancing** | Even traffic distribution | Round robin, Least connections, IP hashing |
| **Caching** | Reduce DB load, faster reads | Store frequent data in memory (Redis) |
| **Database Sharding** | Scale databases horizontally | Split data across multiple DB instances |
| **Database Replication** | High availability, read scaling | Primary writes to replicas automatically |
| **CAP Theorem** | Distributed system trade-offs | Consistency, Availability, Partition Tolerance — pick 2 |
| **Rate Limiting** | Prevent abuse | Token Bucket, Sliding Window algorithms |
| **Consistent Hashing** | Minimizes resharding when nodes change | Used in distributed caches and DHTs |
| **Event-Driven Architecture** | Decoupled microservices | Services communicate via events/messages |

## 6. STAR Method

### 6.1 What is STAR?

**STAR** is a structured framework for answering behavioral interview questions with clarity and impact.

| Letter | Meaning | Guiding Question |
|---|---|---|
| **S** | Situation | What was the context? Set the scene. |
| **T** | Task | What was your responsibility or goal? |
| **A** | Action | What specific steps did YOU take? |
| **R** | Result | What was the measurable outcome? |

**Time allocation:** Situation (10%) → Task (10%) → Action (60%) → Result (20%)

---

### 6.2 Five Detailed STAR Examples

---

#### Q1: "Tell me about a time you faced a challenging technical problem."

**S:** During my final year project, our team's React application started throwing critical memory leak errors after 15 minutes of use in production testing, causing the browser tab to crash.

**T:** As the lead frontend developer, I was responsible for diagnosing and resolving the issue within 48 hours before our college demo.

**A:** I used Chrome DevTools' Memory tab to take heap snapshots before and after rendering. I identified that event listeners attached in `useEffect` were not being cleaned up on component unmount. I systematically audited every useEffect in the codebase, added proper cleanup functions, and wrote a custom `useEventListener` hook to standardize this across all components.

**R:** Memory usage dropped from 800MB after 15 minutes to a stable 90MB throughout the session. The app ran flawlessly during the demo, and I shared this hook as an open-source GitHub Gist that received 200+ stars.

---

#### Q2: "Describe a time you worked in a team with conflict."

**S:** During a hackathon, my 4-person team was split on whether to use React Native or Flutter for our mobile app. Two members strongly preferred each option, and the debate was consuming valuable time.

**T:** As the team lead, I needed to resolve the conflict quickly and get everyone aligned within the first 2 hours of the 24-hour hackathon.

**A:** I proposed a quick 15-minute structured comparison: each side had 7 minutes to list pros and cons specific to our use case. I then reframed the decision around our constraints — all four of us had stronger JavaScript skills, so React Native gave us the highest confidence and speed. I acknowledged the Flutter team's concerns by assigning them the UI design which leveraged their skills regardless of platform.

**R:** We aligned within 30 minutes, built the app in React Native, and submitted with 2 hours to spare. Our team won 2nd place, and the judges specifically praised our smooth UI. The experience taught me that data-driven decisions defuse emotional arguments.

---

#### Q3: "Tell me about a time you showed leadership."

**S:** In my 3rd year, I noticed that many juniors in my department were struggling with DSA basics because there was no structured mentoring program at LPU for first-year CSE students.

**T:** I decided to independently launch a weekly peer mentorship session to bridge this gap.

**A:** I created a 12-week DSA curriculum covering Arrays through Graphs, sourced practice problems from LeetCode and Codeforces, and booked a classroom every Saturday for 2 hours. I recruited 3 senior students to co-mentor and used WhatsApp groups to share resources. I also created a shared Google Sheet to track each student's weekly progress.

**R:** Over one semester, 45 students attended the sessions. 8 of them later cleared placement aptitude rounds at service-based companies. The initiative was recognized by the CSE department and was adopted as an official student club activity in the following year.

---

#### Q4: "Describe a time you failed and what you learned."

**S:** In my 2nd year, I applied for a prestigious national-level hackathon (Smart India Hackathon) with a project idea I was very confident about. I spent two weeks preparing the pitch but only 3 days on the actual prototype.

**T:** My goal was to represent LPU at the national finals and potentially win recognition for the university.

**A:** During the internal selection round, the judges asked deep technical questions about the implementation which I could not answer confidently. My prototype had too many bugs and my demo crashed midway.

**R:** We were not selected. However, I took this as a turning point. I completely reversed my approach: for the next hackathon, I spent 80% of the time on technical depth and 20% on presentation. Six months later, I participated in a 24-hour hackathon, built a fully functional prototype, and won 1st place. The lesson was: **substance before style**.

---

#### Q5: "Tell me about a time you went above and beyond."

**S:** During my internship at a fintech startup, I was assigned to build a simple data export feature for the admin panel — exporting CSV files. The sprint had 5 days allocated.

**T:** My task was straightforward: create a button that downloaded the current data table as a CSV.

**A:** While building the feature, I noticed the data being exported had inconsistent date formats and duplicate entries due to an underlying ETL bug. Though it was outside my ticket's scope, I reported it to my manager, proposed a fix, and offered to implement it if given one extra day. They agreed. I wrote a data cleaning utility and integrated it into the export pipeline, added column header mappings for better readability, and included a progress indicator for large exports.

**R:** The feature shipped 1 day late but was praised in the team retro as one of the most polished deliverables of the sprint. My manager specifically mentioned my initiative in my internship evaluation, which resulted in a PPO (Pre-Placement Offer) consideration at the end of the internship.

### 7.2 "Why This Company?" — Framework

Use the **3R Framework:** Research + Resonate + Role

1. **Research:** Mention something specific about the company (product, culture, tech stack, mission)
2. **Resonate:** Connect it to your personal values or career goal
3. **Role:** Explain how this specific role helps you contribute and grow

**Example (for Microsoft):**
> "I've followed Microsoft's developer ecosystem for years. The shift to open-source with VS Code and .NET has shown me a company that values developer community — something I care deeply about since I contribute to open-source projects myself. The cloud-first vision with Azure aligns with the distributed systems work I've been exploring. This SWE role specifically, working on Azure's core infrastructure, is exactly the kind of high-scale, high-impact work I want to grow into."

### 7.4 "Where Do You See Yourself in 5 Years?"

**Framework:** Short-term (1–2 years) → Medium-term (3–4 years) → Long-term alignment with company

**Example:**
> "In the first two years, I want to deeply understand the engineering challenges at [Company], become proficient in the codebase, and contribute meaningfully to the team. By year 3–4, I see myself taking ownership of a module or feature end-to-end and possibly mentoring junior engineers. In 5 years, I hope to be in a senior technical role where I'm solving complex system problems at scale. I believe [Company]'s environment of continuous learning and impactful engineering challenges is the best place for that growth path."

### 8.2 Pre-Placement Talk (PPT) Strategy

The Pre-Placement Talk is your **first real interaction** with the company. Most students treat it as a formality — smart students treat it as an opportunity.

**Before the PPT:**
- Research the company: products, recent news, tech stack, culture
- Prepare 3–5 intelligent questions about the role, team, or onboarding
- Review the company's LeetCode tag problems and be mentally prepared

**During the PPT:**
- Sit in the first 3 rows — recruiters notice engaged, visible students
- Take notes — this shows professionalism and helps in follow-up questions
- Ask at least 1 thoughtful question — it differentiates you from 200+ other students

**After the PPT:**
- Note down key themes: What did the company emphasize? Speed? Quality? Innovation?
- Use these themes in your interview answers (e.g., if they emphasize ownership, use that word)

### 9.2 Body Language Tips

**Before Entering the Room:**
- Take 3 deep breaths to lower cortisol (stress hormone)
- Stand with shoulders back and head up (power pose for 2 minutes)
- Smile — it signals confidence and warmth

**During the Interview:**
- Maintain natural eye contact (not staring — look away occasionally)
- Lean slightly forward to show engagement
- Nod to show you're actively listening
- Avoid: crossing arms, fidgeting, looking at the floor, speaking too fast
- Mirror the interviewer's energy — if they're formal, match that; if friendly, relax

**When Answering:**
- Pause 2–3 seconds before answering — it shows thoughtfulness, not confusion
- If you don't understand, ask: "Could you clarify what you mean by [X]?"
- Never interrupt the interviewer

---

### 9.3 Technical Setup for Online Interviews

**24 Hours Before:**
- Test your camera, microphone, and internet connection
- Install required software (HackerRank, Coderbyte, Zoom, Teams) and test-run them
- Charge your laptop fully and keep the charger plugged in
- Choose a quiet, well-lit room with a neutral background (or use a virtual background)

**30 Minutes Before:**
- Join the meeting link 5 minutes early
- Have a backup: phone hotspot ready if WiFi drops
- Close unnecessary browser tabs and applications
- Have water nearby (you'll need it)
- Have a physical notepad for rough work (don't type while thinking — it looks distracted)

**During the Online Interview:**
- Look at the **camera**, not your own video feed — this is "eye contact" in online interviews
- Speak clearly and at a moderate pace — audio delays can cause misunderstanding
- If the connection drops: message on WhatsApp/email immediately and rejoin ASAP
- Code in a shared editor (HackerRank, CoderPad) and think out loud while coding

## 10. Mental Health During Placements

### 10.1 Managing Rejection

Rejection is **statistically inevitable** — even top performers face it. The question is not if you'll be rejected, but how you'll respond.

**Immediate After Rejection:**
- Allow yourself to feel disappointed — it's normal and valid
- Do NOT check or compare results with peers immediately
- Do something that resets you: walk, exercise, eat, sleep

**48 Hours After Rejection:**
- Request feedback if the company allows it (many do via email)
- Analyze honestly: Was it DSA? Communication? Attitude? Preparation gap?
- Write it down. Make it concrete. Then make a plan.

**The Mindset Shift:**
> "Each rejection is a rehearsal. Every interview I give, I get better. My job offer is waiting — I just haven't found the right match yet."

**Practical Rejections Tracker (build this in Notion or a Google Sheet):**

| Company | Date | Round Failed | Reason (Self-Analysis) | Action Item |
|---|---|---|---|---|
| Amazon | DD/MM | Technical R1 | Could not solve DP problem | Practice 20 more DP problems this week |

---

### 10.2 Handling Peer Pressure

One of the most psychologically damaging aspects of placement season is **comparison culture**. When your hostel friends start getting offers, it can feel devastating.

**What to do:**
- Remember: placements are **not a race**. Getting an offer in October vs January doesn't define your career.
- Limit social media usage during placement season — LinkedIn can be a double-edged sword.
- Celebrate friends' success genuinely — this is a test of character and builds goodwill.
- Focus on **inputs (daily preparation hours), not outcomes (offer count)**. Outcomes follow consistent inputs.
- If a friend got placed and you didn't — ask them what helped, not to compare packages.

---

### 10.3 Maintaining Routine

A structured routine is your **mental anchor** during the chaos of placement season.

**Key Habits to Build:**
- **Sleep:** 7–8 hours minimum. Sleep deprivation severely impairs problem-solving ability.
- **Exercise:** 30 minutes of any physical activity daily. Even a walk. It's not optional.
- **Meals:** Don't skip meals during placement drives. Hunger tanks your focus and mood.
- **Breaks:** Use the Pomodoro technique — 25 min work, 5 min break. Prevents burnout.
- **Journaling:** Write 3 things you're grateful for every night. Maintains perspective.
- **No all-nighters before interviews:** Sleep is more valuable than one more hour of LeetCode.
